## Одна команда

```bash
/bin/bash -c "$(curl --proto '=https' --tlsv1.2 -fsSL \
  https://raw.githubusercontent.com/talkstream/telegram-ipv6-doctor/v1.0.1/telegram-ipv6-doctor.sh)"
```

Она **только читает и ничего не трогает**: без sudo, без изменений, без телеметрии. На выходе — вердикт.

Хотите сначала посмотреть, что запускаете? Это правильная привычка — и ссылка выше неизменяемая, поэтому
сверить контрольную сумму имеет смысл:

```bash
curl --proto '=https' --tlsv1.2 -fsSLO https://raw.githubusercontent.com/talkstream/telegram-ipv6-doctor/v1.0.1/telegram-ipv6-doctor.sh
shasum -a 256 telegram-ipv6-doctor.sh    # сверьте с контрольной суммой в описании релиза
less telegram-ipv6-doctor.sh
bash telegram-ipv6-doctor.sh
```

## Что именно он различает

У «Connecting…» не одна причина, и неверный фикс делает только хуже. Инструмент проверяет каждый
дата-центр Telegram **по обоим** протоколам — IPv4 и IPv6, — опрашивает независимые контрольные IPv6-цели,
читает счётчики в логах клиента и ищет локальный сетевой фильтр. А затем разводит причины:

| Вердикт | Что это значит | Что делает инструмент |
|---|---|---|
| **IPv6 к Telegram сломан** | IPv6 к дата-центрам Telegram уходит в тайм-аут, IPv4 к ним же работает, а IPv6 к остальному интернету здоров | предлагает точечный фикс ↓ |
| **IPv6 сломан целиком** | IPv6 не работает ни к чему, не только к Telegram | предлагает выключить IPv6 на интерфейсе |
| **мешает локальный фильтр** | Сеть чистая, но файрвол или антивирус (Little Snitch, LuLu…) рвёт сокеты Telegram | **сеть не трогает** — говорит проверить фильтр |
| **блокировка / DPI** | Соединения устанавливаются и сразу рвутся | советует MTProxy и отказывается что-либо «чинить» |
| **петля в клиенте** | Сеть в порядке, циклится сам клиент | советует перезапуск или поддержку Telegram |
| **сеть только на IPv6 (NAT64)** | Пути по IPv4 нет вообще | **запрещает любые изменения** — фикс отрезал бы вас от связи |
| **всё в порядке** | Проблем с сетью не видно | ничего не делает |

Третья строка появилась не из теории. После того как мы починили настоящую проблему с IPv6, тот же Mac снова
начал зависать — и виновата оказалась ночная сборка DPI-файрвола, а вовсе не провайдер. Инструмент, умеющий
винить только IPv6, в этот момент солгал бы. Подробности — в [`docs/FINDINGS.md`](docs/FINDINGS.md).

## Фикс

```bash
/bin/bash -c "$(curl -fsSL .../v1.0.1/telegram-ipv6-doctor.sh)" _ fix     # символ «_» обязателен: без него bash съедает первый аргумент
```

**По умолчанию — точечный.** Инструмент ставит reject-маршруты **только** на те IPv6-префиксы Telegram,
поломку которых он только что доказал на вашей машине:

```
sudo route -n add -inet6 2001:b28:f23d::/48 ::1 -reject
```

После этого соединения к Telegram по IPv6 падают **мгновенно**, а не висят — клиент сразу уходит на IPv4,
и **весь остальной IPv6 продолжает работать** (сайты, почта, всё прочее). Маршруты исчезают при перезагрузке,
а команда `revert` убирает их немедленно.

Грубая альтернатива (`--mode ipv6-off`, отключение IPv6 на интерфейсе) — для вердикта «IPv6 сломан целиком».

### Он отказывается работать, когда работать нельзя

- нет рабочего пути по **IPv4** к Telegram → отказ (фикс отрезал бы вас полностью);
- сеть **только на IPv6 / NAT64** → отказ даже с `--yes`;
- вердикт не тот, ради которого фикс написан → отказ;
- активен **VPN или прокси** → отказ (данным нельзя доверять);
- локальный **фильтр рвёт сокеты** → отказ (чинить надо фильтр, а не сеть).

Каждую привилегированную команду инструмент печатает **до** запуска. С флагом `--dry-run` он печатает их
и выходит. Состояние он сохраняет до любых изменений, а `revert` возвращает его — с проверкой результата,
а не на веру.

## Приватность

Телеметрии нет вообще. Единственный сетевой трафик — сами пробы, и за этим следит
[тест](tests/doctor.bats), который падает, если скрипт хотя бы попытается вызвать `curl` в рантайме.
`report` собирает только то, что в белом списке: версия macOS, версия клиента, задержки, счётчики, вердикт.
Ни имён хостов, ни адресов, ни SSID, ни имён аккаунтов, ни путей.

## Как он устроен

- **Bash 3.2** — та версия, что идёт в штатном macOS. Ноль зависимостей: `nc`, `route`, `scutil`, `perl` есть в системе.
- IPv6-префиксы Telegram — **константа в коде**, взятая из [официального списка сетей Telegram](https://core.telegram.org/resources/cidr.txt) и сверенная с живыми BGP-анонсами. В рантайме список не загружается никогда: подменённый источник отправил бы чужие сети в blackhole — **с правами root**. За свежесть отвечает еженедельный CI-джоб: он сверяет официальный список с константой и открывает PR для человека.
- Reject ставится только на префиксы, **доказанно сломанные на вашей машине**. Ничего «на всякий случай».
- 18 офлайн-тестов (bats + фейковые утилиты) покрывают каждый вердикт, каждый отказ и приватность отчёта.

## Аналоги

Точных аналогов нет: есть зеркала CIDR-списка Telegram, есть универсальные тестеры Happy Eyeballs, но нет
инструмента под macOS, который проверяет дата-центры Telegram по обоим протоколам и умеет действовать по
результату. Знаете такой — откройте issue.
